Section 1

Foundations - Big-O and Complexity

Before diving into specific data structures, you need a shared language for performance: Big-O notation, time vs. space complexity, and a feel for common patterns.

1.1 What Are Data Structures and Algorithms?

A data structure is a way to organize and store data so it can be used efficiently.

  • Examples: arrays, linked lists, hash tables, trees, graphs.

An algorithm is a step-by-step procedure for solving a problem.

  • Examples: binary search, merge sort, Dijkstra's algorithm.

The same problem can be solved with many different data structures and algorithms. Some are dramatically faster or more memory-efficient than others. Big-O notation gives us a way to talk about that efficiency, independent of hardware or constant factors.

1.2 Big-O Notation (Time Complexity)

Big-O describes how an algorithm's running time grows as the input size n grows.

  • O(1) - constant time: roughly the same time no matter how big n is.
  • O(n) - linear time: work grows proportionally with n.
  • O(n²) - quadratic time: often from nested loops over the same data.
  • O(log n) - logarithmic: repeatedly cutting the problem in half (for example, binary search).
  • O(n log n) - common for efficient sorting (merge sort, quicksort average, heapsort).

We do not care about exact milliseconds; we care about how the curve behaves as n gets large.

1.3 Space Complexity

Space complexity measures how much extra memory an algorithm uses beyond the input itself.

  • Creating a new array/list of size n is typically O(n) extra space.
  • Reading and printing values without storing extra structures is often O(1) extra space.
Rule of thumb: Time complexity is about "how many steps," space complexity is about "how much extra stuff" you allocate while running.

1.4 Code Examples in Multiple Languages

Below, each example appears in six languages: C, C++, Java, JavaScript, C#, and Python. Use the language tabs to switch between them.

1.5 Example 1 - O(n): Print All Elements

This function loops through an array/list and prints each element once.

Example 1 Print every element once 
// C
#include <stdio.h>

void ex1(int arr[], int n) {
    for (int i = 0; i < n; i++) {
        printf("%d\n", arr[i]);
    }
}
// C++
#include <iostream>
#include <vector>
using namespace std;

void ex1(const vector<int>& arr) {
    for (int x : arr) {
        cout << x << endl;
    }
}
// Java
void ex1(int[] arr) {
    for (int x : arr) {
        System.out.println(x);
    }
}
// JavaScript
function ex1(arr) {
    for (const x of arr) {
        console.log(x);
    }
}
// C#
void Ex1(int[] arr) {
    foreach (var x in arr) {
        Console.WriteLine(x);
    }
}
# Python
def ex1(arr):
    for x in arr:
        print(x)
Time: O(n) — one pass over n elements.  |  Extra space: O(1).

1.6 Example 2 - O(n²): Nested Loops Over the Array

This function prints all pairs (arr[i], arr[j]) with j starting at i.

Example 2 Print all ordered pairs (i, j) with j ≥ i 
// C
#include <stdio.h>

void ex2(int arr[], int n) {
    for (int i = 0; i < n; i++) {
        for (int j = i; j < n; j++) {
            printf("%d, %d\n", arr[i], arr[j]);
        }
    }
}
// C++
#include <iostream>
#include <vector>
using namespace std;

void ex2(const vector<int>& arr) {
    for (int i = 0; i < arr.size(); i++) {
        for (int j = i; j < arr.size(); j++) {
            cout << arr[i] << ", " << arr[j] << endl;
        }
    }
}
// Java
void ex2(int[] arr) {
    for (int i = 0; i < arr.length; i++) {
        for (int j = i; j < arr.length; j++) {
            System.out.println(arr[i] + ", " + arr[j]);
        }
    }
}
// JavaScript
function ex2(arr) {
    for (let i = 0; i < arr.length; i++) {
        for (let j = i; j < arr.length; j++) {
            console.log(arr[i], arr[j]);
        }
    }
}
// C#
void Ex2(int[] arr) {
    for (int i = 0; i < arr.Length; i++) {
        for (int j = i; j < arr.Length; j++) {
            Console.WriteLine($"{arr[i]}, {arr[j]}");
        }
    }
}
# Python
def ex2(arr):
    for i in range(len(arr)):
        for j in range(i, len(arr)):
            print(arr[i], arr[j])
Time: O(n²) — nested loops over n.  |  Extra space: O(1).

1.7 Example 3 - O(1): Constant Time Access

This function prints the first and last elements (assuming the array is non-empty).

Example 3 Read first and last element 
// C
#include <stdio.h>

void ex3(int arr[], int n) {
    if (n > 0) {
        printf("%d %d\n", arr[0], arr[n - 1]);
    }
}
// C++
#include <iostream>
#include <vector>
using namespace std;

void ex3(const vector<int>& arr) {
    if (!arr.empty()) {
        cout << arr.front() << " " << arr.back() << endl;
    }
}
// Java
void ex3(int[] arr) {
    if (arr.length > 0) {
        System.out.println(arr[0] + " " + arr[arr.length - 1]);
    }
}
// JavaScript
function ex3(arr) {
    if (arr.length > 0) {
        console.log(arr[0], arr[arr.length - 1]);
    }
}
// C#
void Ex3(int[] arr) {
    if (arr.Length > 0) {
        Console.WriteLine($"{arr[0]} {arr[^1]}");
    }
}
# Python
def ex3(arr):
    if arr:
        print(arr[0], arr[-1])
Time: O(1) — a constant number of operations.  |  Extra space: O(1).

1.8 Example 4 - O(n) + O(n) = O(n)

Two separate loops each run n times. Together they are still linear.

Example 4 Two passes over the same array 
// C
#include <stdio.h>

void ex4(int arr[], int n) {
    for (int i = 0; i < n; i++) {
        printf("%d\n", arr[i]);
    }
    for (int i = 0; i < n; i++) {
        printf("%d\n", arr[i]);
    }
}
// C++
#include <iostream>
#include <vector>
using namespace std;

void ex4(const vector<int>& arr) {
    for (int x : arr) {
        cout << x << endl;
    }
    for (int x : arr) {
        cout << x << endl;
    }
}
// Java
void ex4(int[] arr) {
    for (int x : arr) {
        System.out.println(x);
    }
    for (int x : arr) {
        System.out.println(x);
    }
}
// JavaScript
function ex4(arr) {
    for (const x of arr) {
        console.log(x);
    }
    for (const x of arr) {
        console.log(x);
    }
}
// C#
void Ex4(int[] arr) {
    foreach (var x in arr) {
        Console.WriteLine(x);
    }
    foreach (var x in arr) {
        Console.WriteLine(x);
    }
}
# Python
def ex4(arr):
    for x in arr:
        print(x)
    for x in arr:
        print(x)
Time: O(n) — two linear passes still scale linearly.  |  Extra space: O(1).

1.9 Example 5 - O(n) Time, O(n) Extra Space

This function builds a new array/list where every element is doubled.

Example 5 Allocate and fill a new array 
// C
#include <stdlib.h>

int* ex5(int arr[], int n) {
    int* newArr = (int*)malloc(sizeof(int) * n);
    if (!newArr) return NULL; // handle allocation failure

    for (int i = 0; i < n; i++) {
        newArr[i] = arr[i] * 2;
    }
    return newArr; // caller should free(newArr)
}
// C++
#include <vector>
using namespace std;

vector<int> ex5(const vector<int>& arr) {
    vector<int> res;
    res.reserve(arr.size());
    for (int x : arr) {
        res.push_back(x * 2);
    }
    return res;
}
// Java
int[] ex5(int[] arr) {
    int[] res = new int[arr.length];
    for (int i = 0; i < arr.length; i++) {
        res[i] = arr[i] * 2;
    }
    return res;
}
// JavaScript
function ex5(arr) {
    return arr.map(x => x * 2);
}
// C#
int[] Ex5(int[] arr) {
    return arr.Select(x => x * 2).ToArray();
}
# Python
def ex5(arr):
    return [x * 2 for x in arr]
Time: O(n) — one pass to build the new array.  |  Extra space: O(n) — new array stores n items.

A data structure has two sides:

  • Shape: how the data is laid out in memory (for example, contiguous, linked, tree).
  • Operations: what you can do efficiently (for example, insert, delete, search, iterate).

For example, both an array and a linked list store a sequence of elements, but:

  • Arrays give O(1) random access by index, but inserting in the middle is O(n).
  • Singly linked lists give cheap O(1) insertion at the front, but random access is O(n).

An algorithm is a concrete sequence of steps that operates on some data structure to solve a task. Sorting, searching, traversing a tree, or computing a shortest path are all algorithms.

The same problem can be solved with different data-structure and algorithm combinations. Choosing the right structure often changes which algorithms are even possible or efficient. That is why interviews and system design conversations keep asking "what data structure would you use here?".

Big-O is a way to describe an upper bound on running time as the input size n grows. Formally, f(n) is O(g(n)) if there are constants C and n0 such that:

f(n) ≤ C · g(n) for all n ≥ n0.

In practice, we use it as a coarse "growth rate" label:

  • O(1): fixed amount of work, like reading a single array element.
  • O(n): work proportional to the number of items, like a single loop.
  • O(n²): work proportional to n × n, often double loops over the same data.
  • O(log n): work proportional to the number of times you can cut the problem in half.

Big-O deliberately ignores:

  • Constant factors: O(2n) and O(100n) are both O(n).
  • Lower-order terms: O(n² + n) is still O(n²).

This lets you compare algorithms across machines and implementations. If one algorithm is O(n) and another is O(n²), the O(n²) approach eventually loses badly as n grows, even if it is faster for tiny inputs.

You will also sometimes see:

  • Θ (theta): tight bound (both upper and lower).
  • Ω (omega): lower bound (at least this fast).

Interviews almost always use Big-O in the loose upper-bound sense: "In the worst case, this algorithm runs in O(n log n) time."

When we talk about space complexity, we separate:

  • Input: the memory already used to store the data you are given.
  • Extra space: any additional memory the algorithm allocates while it runs.

Some common patterns:

  • Algorithms that only keep a few counters or pointers usually use O(1) extra space.
  • Algorithms that build new arrays, lists, or maps proportional to n usually use O(n) extra space.
  • Recursive algorithms grow the call stack: a recursion depth of d can mean O(d) stack space.

An algorithm is often called in place if it uses O(1) extra space beyond a small number of variables and the call stack, even if it modifies the input array or list.

You will frequently trade time and space against each other:

  • Using a hash table can turn an O(n²) algorithm into O(n), but you pay O(n) extra memory.
  • Recomputing values instead of caching them may save memory but cost more time.

In interviews, say which kind of space you are counting: "This solution runs in O(n) time and O(1) extra space."

Example 1 touches each element exactly once and does a constant amount of work per element (a print call). If the array doubles in size, the number of loop iterations also doubles.

We can describe the total work as something like:

work(n) = a · n + b

for some constants a and b. In Big-O notation, that is O(n) because linear growth dominates the constant overhead as n becomes large.

Example 2 has a loop inside a loop. If there are n elements:

  • When i = 0, j runs from 0 to n - 1 (about n iterations).
  • When i = 1, j runs from 1 to n - 1 (about n - 1 iterations).
  • ...
  • When i = n - 1, j runs only once.

The total work is roughly:

(n) + (n - 1) + (n - 2) + ... + 1 = n(n + 1)/2

That formula is on the order of n², so we say the algorithm is O(n²).

Example 3 reads at most two elements: the first and the last. It does not matter if the array has 10 elements or 10 million elements, the algorithm still performs a fixed number of operations.

That is what "constant time" means: the running time does not grow with n.

Example 4 runs two separate loops, each O(n):

  • The first loop prints all elements once.
  • The second loop prints all elements once again.

The total work is something like:

work(n) = a · n + a · n = 2a · n

Big-O ignores constant factors like 2, so we still call this O(n).

Example 5 builds a new array or list that holds every element of the original, doubled. The new structure stores n items, so its size grows linearly with n.

That gives:

  • Time: O(n) to visit each element and write it once.
  • Extra space: O(n) for the new array or list that holds all doubled values.

If you needed to save space, you might consider updating the original array in place instead, trading away the ability to keep the old values.
